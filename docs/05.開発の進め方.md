# AI共創基盤 - 開発の進め方

## Phase 0: 契約定義（先にやる）

全サービスの接続面を先に決める。

- OpenAPI specs（全サービスのAPI仕様）
- DBスキーマ（Drizzle マイグレーション）
- Docker Compose（PostgreSQLのみ）+ Makefile
- モノレポ骨格（各サービスのディレクトリ・初期設定ファイル）

## Phase 1: サービス開発（依存順に段階的に進める）

全てローカルで開発。Honoは通常のNode.js HTTPサーバーとして動作するため、Lambda等のエミュレーションは不要。

### Phase 1a: 基盤（並列: 1〜2）

```bash
git worktree add ../proj-auth   feature/auth
```

- DB schema + auth サービス
- 他の全サービスの前提となるため最初に着手

### Phase 1b: データ・処理層（並列: 2）

```bash
git worktree add ../proj-feed   feature/feed
git worktree add ../proj-notif  feature/notification
```

- feed, notification を並行開発
- 前提: auth完了、DBスキーマ確定

### Phase 1c: オーケストレーション層（並列: 2）

```bash
git worktree add ../proj-bff    feature/bff
git worktree add ../proj-ai     feature/ai
```

- bff, ai を並行開発
- 前提: feed/notification のAPIが安定

### Phase 1d: クライアント層（並列: 2）

```bash
git worktree add ../proj-web    feature/web
git worktree add ../proj-cli    feature/cli
```

- web, cli を並行開発
- 前提: bff のAPIが安定

## Phase 2: 結合テスト

各ブランチをmainにマージ → `make dev` で全サービスをローカル起動して結合テスト。

## Phase 3: デプロイ

結合テスト完了後、CDKを作成して本番環境を構築する。
Honoのコードは同一で、Lambda adapterをエントリポイントとして追加するだけ。

```bash
make deploy       # CDKデプロイ（Lambda, EventBridge等）
```

| コンポーネント | デプロイ先 |
|---|---|
| auth, bff, feed, notification | AWS Lambda（Hono Lambda adapter） |
| ai | AgentCore Runtime |
| web | Vercel |
| cron（feed:fetch, ai:digest） | EventBridge Scheduler |
| DB | Supabase |

## 決定事項

- [x] AIデプロイ先: AgentCore Runtime（サーバーレス、従量課金）
- [x] Bedrockで使用するモデル: Claude Haiku（低コスト、要約タスクに十分）
- [x] 認証状態管理: Web はセッション（httpOnly cookie、PostgreSQL保存）、CLI/MCP は Bearerトークン
- [x] RSS以外のブックマーク: WebUIからURL入力で登録
- [x] デプロイ先: TSサービス→AWS Lambda、web→Vercel、DB→Supabase、cron→EventBridge
